标题：Markdown 可能要给 AI 让位了，我做完一个 3191 行的 HTML 简历站才搞明白

作者：炸牛奶
日期：2026-05-20

---

故事是这样的。

这两天 Anthropic 自己博客上发了篇东西，标题挺嚣张的，叫 The unreasonable effectiveness of HTML，翻成中文就是「HTML 好用得有点离谱」。作者是 Claude Code 团队的，核心就一句话，他们现在让 Claude 输出东西的时候，不再首选 Markdown 了，改用 HTML。

我看到这篇的时候，刚关掉自己的 VS Code。屏幕上停着的，是一份 3191 行的 HTML 文件，prototype-v2.4.html，是我最近两个礼拜跟 Claude Code 来回七八次迭代出来的个人简历网站。

就那一瞬间，我有一种很奇怪的感觉。

不是「卧槽 Anthropic 跟我想到一块儿去了」那种激动，而是更冷静的、更后知后觉的一种回神，怎么说呢，就好像你已经在某条路上走了好几个月，每天就这么走着也没多想，然后有一天路边突然立了一块路牌，上面写着这条路的名字，你才意识到「噢，原来我走的这条路有名字啊」。

这就是这篇文章想聊的事儿。

【配图 1：文章封面图。可以是 Anthropic 那篇博客的标题截图 + 你自己 VS Code 里 prototype-v2.4.html 的局部截图，做一个对比拼图。或者直接截 Claude 那篇博客头图。】

我不是一个程序员，我做的是产品和设计。我做个人网站这个事，从头到尾没写过一行手敲的 HTML，全是跟 Claude Code 协作出来的。但回头看，我发现一个挺有意思的现象，我跟 AI 协作的输出物，已经几乎全是 HTML 了，Markdown 在我这边的使用率正在以一种我自己没察觉的速度往下掉。

然后 Anthropic 的人也是同样的判断。

所以今天想聊聊这个事儿，AI 时代的内容输出形态，可能真的要换一茬了。

---

先聊聊 Markdown 是怎么来的。

Markdown 的设计者是 John Gruber，2004 年搞出来的，初衷特别朴素，他想要一种简单的纯文本格式，让作者写起来不别扭，转成 HTML 之后又长得不能看。注意他当时考虑的主语，是「作者」，是「人」。

从 2004 一路到 2022 大模型起飞之前，Markdown 都是为人服务的。程序员写 README，写文档，写博客。我们做产品的写 PRD，写设计稿规范。简洁清爽，懒人福音。

然后 ChatGPT 来了。

从 ChatGPT 开始，所有的大模型对话默认输出格式都是 Markdown。你问它什么，它甩你一坨 #、##、###，然后加粗加列表加表格。整整三年，我们都习惯了 AI 用这种方式跟我们说话。

很多朋友可能没意识到，这三年发生了一件特别隐蔽的事，Markdown 的使用主语，已经从「人写给人看」悄悄变成了「AI 写给人看」。

当主语换了，介质就不一定合适了。

这个判断不是我先有的，是 Anthropic 那篇博客点醒我的。Claude Code 团队的人原话是这么说的，「随着 agent 越来越强，我发现 Markdown 变成了一个越来越受限的格式」。然后他列了几条具体的痛点，每一条我读完都想说，对对对，我也是。

---

第一条，超过 100 行的 Markdown 我读不下去。

说真的，这条戳得我心口直疼。

我做简历站的时候，第一版让 Claude 帮我出了一份「核心 AI 工作流设计规范」，让它把我平时用 AI 的流程结构化梳理出来，方便我自己看，也方便招聘官看到的时候能 get 到我的工作方式。

它给我吐了一份 .md 文件，三百多行。

我打开看了一眼，关掉了。

我跟你说真的，三百多行的 Markdown 在那里，所有 # 看起来一样长，所有 - 看起来一样平，密密麻麻的文本像一面墙糊在我脸上。我作为这份文档的「需求提出者」，我自己都看不进去。

那我怎么指望招聘官看完？

那一刻我做的事情很简单，我跟 Claude 说，把这份规范改成一个 HTML 页面，要有视觉层级，五个阶段用并列的卡片，每个阶段的产出物给我画个示意图，能展开能收起。

它给我吐了一个 HTML 文件，我用浏览器打开，那一秒我才看懂自己原来想表达什么。

【配图 2：左右对比截图。左边是 300 行 Markdown 文档的截屏（密密麻麻一面墙），右边是改成 HTML 之后的可视化页面（核心 AI 工作流 5 阶段卡片）。你的 prototype-v2.4.html 里就有这块，截就行。这张图是全文最重要的一张视觉对比，建议自己出。】

Markdown 它不是写得不好，它是没法承载这种东西。AI 现在能一口气生成的内容，已经远远超过了 Markdown 这套上世纪的语法能承载的密度。当 AI 想跟你聊一个有 5 个并行模块、每个模块下面 3 层子结构、还需要图示和颜色区分的事情的时候，Markdown 唯一能做的就是把它压扁成一面墙。

---

第二条，AI 在 Markdown 里强凑可视化的样子，真的有点心酸。

这个观察可能写代码的朋友更有共鸣。你让 ChatGPT 或者 Claude 在对话里画一张架构图，它会怎么办？

它会用 ASCII 字符给你拼。

┌─────────┐    ┌─────────┐
│  用户   │ -> │ 网关层  │
└─────────┘    └─────────┘

这种东西。

更骚的是，Anthropic 那篇博客里提到，他们发现 Claude 会用 unicode 字符去「估算颜色」。我看到这条的时候笑出声，但笑完又有点心疼。

这是什么意思呢，就是你让 AI 在 Markdown 里画个有颜色的图，它做不到，因为 Markdown 没有颜色这个概念。但是它又想表达「这一块是红色的、那一块是蓝色的」，怎么办？

它就会用一些特殊字符，比如方块 ■、圆点 ●，配上不同 Unicode 区段里那些自带「色感」的符号，给你拼一个能让你「感觉到颜色」的近似图。

真的太离谱了。

这就好比你给一个画家一支只能画黑白的笔，让他画一幅彩色的画。他画不出来，但他足够努力，他会用各种笔触的疏密、各种斜线交叉的角度，去暗示「这块应该是红的」「那块应该是蓝的」。你看着甚至能猜出来他想表达的颜色。

但你看着会很难受。

因为你知道，给他一支彩色的笔就行了，根本不用这么扭曲。

HTML 就是那支彩色的笔。SVG、CSS、行内样式、表格、可折叠的 details 标签，AI 想表达的所有那些 Markdown 表达不了的层次、颜色、空间关系，HTML 里全都有现成的。它不需要再扭曲了。

【配图 3：AI 用 ASCII / emoji / unicode 拼图的截图。可以直接让 Claude 帮你生成一个「请用纯 Markdown 画一张三层架构图」的对话截图，截下来配上吐槽，效果极强。建议自己出，自己跟 Claude 现造一段对话就行。】

我做我那个简历站的时候，里面有 82 处 inline 的 HTML/CSS 小图，有的是工作流的流程图，有的是工具栈的并列卡片，有的是 Vibe Coding 小项目的状态徽章。这些东西如果用 Markdown 去表达，要么就是一堆 ASCII 字符（丑），要么就是塞一堆图片（重，加载慢，还得管理资源），要么就是干脆放弃，写一段「这里是一个示意图，请脑补」（lol）。

HTML 让 AI 终于不用脑补了。

【配图 4：你简历站里那 82 处 inline 图的拼接截图，证明「HTML 能让 AI 真正画出来」。建议截 prototype-v2.4.html 里「核心 AI 工作流」5 阶段那块，或者 AI toolkit 那块。自己出。】

---

第三条，HTML 能直接发链接，Markdown 还得当附件。

这条对内容创作者、产品经理、设计师，杀伤力最大。

你想想看这个场景。你跟 AI 协作产出了一份很扎实的东西，比如一份产品分析报告、一份 PR 评审意见、一份调研记录。你想发给同事看，或者发给老板看，或者像我这样，发给招聘官看。

你打开微信，准备发。

如果是 .md 文件，对方接收到之后，他得有一个能打开 markdown 的工具。可能是 VS Code，可能是 Typora，可能是 Notion。他得先下载这个文件到本地，再用工具打开。中间任何一个环节卡住，他就放弃了。

而且 .md 文件在不同的渲染器下，长得还不一样。你在 Typora 里看着挺美的，到了 GitHub 上可能 emoji 都丢了。

你想想看这有多累。

如果是 .html 文件呢？把它丢到任何一个静态托管服务上，GitHub Pages 也行 Netlify 也行 Vercel 也行，你就得到一个链接。对方点开，浏览器直接渲染，手机电脑都能看，响应式自适应。

我做的那个简历站就是一个 HTML 文件，挂在 GitHub Pages 上。我跟招聘官发的从来都是链接，没人需要下载任何东西。打开链接两秒钟，我的整个作品集、工作流、写作、jd 匹配分析模态框，全都在他眼前展开。

【配图 5：你的简历站在手机和电脑上的双屏截图，证明响应式 + 链接即开即看。自己出。截一个 Mac Safari 的窗口 + iPhone 屏幕截图拼一起。】

这是 Markdown 给不了的体验。坦率的讲，公众号发文章其实也是这个逻辑，你点开链接就能看，不用下载任何东西。公众号的产品力之所以这么强，很大一部分就是因为它把「输出 + 分发」这一步打通了，让普通人也能享受到 HTML 时代的便利。

而 AI 时代的所有输出，都应该是这样的。Claude 帮你写了个报告，写完直接给你一个链接，你转发就完事儿了。这种顺滑感，是 Markdown 永远做不到的。

---

第四条，HTML 能让你跟内容互动，这条最让我兴奋。

Anthropic 博客里举了个例子，让我看完愣了很久。

他说，「我有时候想跟 Claude 一起调一个 UI 动画，但是用文字描述太累了。这时候我就让 Claude 直接给我做一个 HTML 文件，里面带几个 slider 和选项，让我自己拖着试，然后给我一个『复制参数』按钮，把我调好的参数复制到剪贴板，我再粘贴回 Claude 那边接着改。」

读到这一段我直接呆住了。

你想想看这是什么？这是 AI 给你「现造」了一个一次性的小工具，专门服务于你当下这个问题。用完即抛。

这种用法，Markdown 是从根本上做不到的。Markdown 永远是只读的，它就是一段文字躺在那儿。HTML 不一样，HTML 里有 JavaScript，有按钮，有滑块，有 input 框。AI 给你产出的不再是「死的报告」，而是「活的工具」。

【配图 6：一个带 slider 的 HTML artifact 演示截图。这张可以让 Claude 帮你现场做一个，比如「做一个调整按钮颜色的 HTML，带 RGB 三个滑块和复制按钮」，截屏。或者直接截你简历站里的 JD 匹配模态框那个交互。建议自己跟 Claude 现造一个，效果最炸。】

我自己做简历站的时候，写了一个特别小的玩意儿。我有一个「JD 匹配度分析」的功能，就是招聘官点开模态框，可以看到我针对这个岗位的具体匹配分析。这个 modal 本身就是 HTML + 一点 JS 拼起来的。如果是 Markdown，这种「点开看详情」根本不存在。要么全摊开，要么全藏起来，没有「按需展开」这个中间状态。

我跟你说，一旦你尝过 HTML 的甜头，你就再也回不去 Markdown 了。

这种感觉，特别像第一次用了 iPad 之后再回去用纸质画板，你能用，但你心里知道你在凑合。

---

说到这块，我得停一下，承认一下对立面。

不是说 Markdown 不行了，Markdown 是不行。

Markdown 在「人写给 AI」这一侧，还是无可替代的。

我每天都在写 Markdown。我写 prompt 的时候是 .md，我配置 CLAUDE.md 的时候是 .md，我写每天的工作记录是 .md，我写这篇文章的初稿在 Notion 里也是 Markdown。

因为对人来说，Markdown 还是最低摩擦的写作格式，没有之一。

你想想看，让我现在直接用 HTML 写这篇文章，每段开头都要敲 <p>，强调要 <strong>，我会疯。Markdown 的语法是为「人在键盘上一边想一边敲」设计的，这个场景下它是最优解。

所以我自己的判断是这样的，未来不是「HTML 取代 Markdown」，而是这两者会各自退到自己最合适的那一侧。

Markdown 守住「人写给 AI」，因为人需要低摩擦的输入语法。

HTML 顶上「AI 写给人」，因为人需要高信息密度的输出体验。

这是一个分工的故事，不是一个替代的故事。

【配图 7：一张分工示意图。左边「人 → Markdown → AI」，右边「AI → HTML → 人」，中间一条分界线。这张可以让 Claude 帮你画，prompt 给它就是这个意思。建议自己出，让 AI 帮画。】

---

回到我自己这条线。

我现在再回头看自己做简历站的这两个礼拜，越看越觉得有意思。

我从头到尾没有「主动选择 HTML」过。我没有在某个时刻坐下来认真想过「这次我要用 HTML 而不是 Markdown」。我就是想做个简历站，简历站默认是网页，网页默认是 HTML，仅此而已。

但回头复盘，从这件事开始，我跟 AI 的协作模式悄悄变了。

第一周我让 Claude 帮我画核心 AI 工作流的图，画完之后我意识到 inline HTML/CSS 比贴图片好太多。第二周我加了「写作」板块，我意识到每篇文章本身就该是一个独立的 HTML 页面，而不是一个塞在主页里的 Markdown blob。这周我加了 JD 匹配模态框，我意识到 HTML 的交互性让简历从一个「死页面」变成了「活页面」。

每一步都不是我策划好的。我就像在玩游戏，每解锁一个新关卡，就发现 HTML 又多了一种用法。

愚钝如我，是直到 Anthropic 那篇博客出来，我才反应过来，这一整条路径，是 AI 时代正在发生的一件大事。

---

说远一点。

知识的载体一直在变。

古代是羊皮卷和竹简，承载手抄的文字，主语是「抄写员写给少数读书人」。

古登堡之后是印刷书，承载排版后的文字，主语是「作者写给大众读者」。

互联网之后是网页，承载多媒体的内容，主语是「内容创作者写给浏览者」。

每一次变化，介质都是为新的主语而生的。

我有时候觉得，我们现在正站在又一个临界点上。AI 时代的内容输出，主语第一次变成了「AI 写给人」，第一次出现了一个「写作者」具备无限可视化能力、无限交互能力、无限信息密度的场景。这种场景下，Markdown 这种「为人手敲设计的简化语法」，自然就显得局促了。

HTML 不是新东西，它已经 30 多岁了。但它从来没有像现在这样，被一个「能在 5 秒内写出 3000 行 HTML 还能自己 debug 的写作者」激活过。

HTML 没变，变的是 HTML 的「写作者」。

过去 30 年，HTML 一直是给人手敲的，所以它一直是那个「写起来麻烦但渲染起来漂亮」的家伙。现在 HTML 的写作者变成了 AI，它的麻烦不再是问题，它的漂亮成了主菜。

这是一个被低估的范式信号。

---

最后说回我自己。

你现在读到这篇文章，如果你是在我的个人网站上看的，那它本身就是一个 HTML 文件，跟它旁边那些我做的其他作品并排躺着。如果你是在公众号里看的，这篇文章的「原版」也在我网站上挂着，链接放在文末。

这个动作本身，就是这篇文章想说的事儿。

我没有把这篇文章先写成 Markdown 再转去公众号。我是先用 Claude 跟我一起，把整篇文章直接写成了一个 HTML 文件，因为我希望它将来能在我网站上长得像个「作品」而不是「博客 post」。

说实话我也还在摸索，这种「直接写 HTML 长文」的工作流到底好不好，会不会有我现在还没意识到的坑。可能有些想法还不成熟。但我自己的感受是，这是一条值得走下去的路。

如果你也在用 Claude Code 或者 Cursor 跟 AI 协作做东西，下次让它给你产出文档的时候，可以试试把 prompt 改一下，从「帮我写个 .md 总结一下」改成「帮我做个 HTML 文件总结一下」。然后你自己用浏览器打开看看。

那一秒可能会有点惊喜。

磨平一些信息差，从一个 HTML 文件开始。
